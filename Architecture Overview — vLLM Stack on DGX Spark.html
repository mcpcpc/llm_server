vLLM Stack -- Architecture Overview

Self-hosted LLM serving on NVIDIA DGX Spark · Qwen3.6-35B-A3B-FP8 · Docker Compose

System Architecture

All services orchestrated via Docker Compose · GPU access via deploy.resources.reservations

Components

vLLM

Inference Engine

Serves the model with an OpenAI-compatible REST API. Handles batching, KV caching, speculative decoding, and GPU memory management. Exposes /v1/chat/completions on port 8000 (internal only).

LiteLLM

Proxy & Auth Gateway

Sits in front of vLLM. Manages per-user API keys, tracks token usage, enforces budgets, and provides a management UI. All client traffic passes through here on port 4000.

PostgreSQL

Persistence Store

Stores LiteLLM's API keys, user records, and usage logs persistently across restarts. Without this, keys and usage history would be lost on container restart.

Qwen3.6-35B-A3B-FP8

The Model

A 35B Mixture-of-Experts coding model with only 3B active parameters per token. Pre-quantized to FP8. Runs at 40% GPU memory utilization, leaving the remainder for the KV cache at 262K context. Uses MTP speculative decoding with 2 tokens for throughput gains.

Request Flow

1

Client sends request

continue.dev or Pi Agent sends a POST /v1/chat/completions request to LiteLLM on port 4000, with a per-user Bearer token and model name qwen3.6-35b-a3b.

2

LiteLLM authenticates & logs

LiteLLM validates the API key against PostgreSQL, records the request metadata (user, tokens, timestamp), enforces any budget limits, then forwards the request to vLLM internally.

3

vLLM processes the request

vLLM schedules the request, checks the prefix cache for reuse, and runs inference on the GPU. MTP speculative decoding drafts 2 tokens ahead per step to increase throughput.

4

Tokens streamed back

Generated tokens are streamed back through LiteLLM to the client in real time. LiteLLM records final token counts in PostgreSQL for usage tracking.

Key vLLM Settings

deploy.resources.reservations Docker Compose native GPU reservation. Requests all NVIDIA GPUs with CUDA capability -- the recommended approach for Compose v2, replacing the older runtime: nvidia.
--gpu-memory-utilization 0.4 Reserves 40% of GPU memory for model weights, leaving the remaining 60% for the KV cache -- important for sustaining the full 262K context window.
--max-model-len 262144 Full 262K token context window -- suitable for large codebases and long agentic sessions.
--speculative-config mtp (2 tokens) Multi-Token Prediction speculative decoding -- predicts 2 tokens ahead per step to improve throughput without requiring a separate draft model.
--enable-prefix-caching Caches KV state for repeated prompt prefixes (e.g. system prompts). Reduces latency on subsequent requests from the same user or tool.
--enable-auto-tool-choice Enables automatic tool/function calling detection, required for agentic workflows in Pi Agent and continue.dev.
--tool-call-parser qwen3_coder Qwen3-specific parser for extracting structured tool call outputs from model responses.
--reasoning-parser qwen3 Parses Qwen3's chain-of-thought reasoning tokens, keeping thinking and final response content correctly separated.
--mamba-cache-mode align Aligns Mamba state cache layout for the hybrid attention/SSM architecture used in Qwen3.6.

User Key Management

Per-user API keys are created via the LiteLLM API using the master key. Each key is tied to a user ID, enabling per-user usage tracking and optional budget enforcement.

POST /key/generate Create a new user key. Supply user_id, key_alias, and optionally max_budget + budget_duration.
GET /key/list List all active keys and their usage statistics.
POST /key/delete Revoke a key immediately. The user's next request will be rejected.
LiteLLM UI · :4000/ui Web dashboard for key management, usage graphs, and per-user token breakdowns. No extra tooling required.
